**Fix sequence:**

3. Add a timeout (e.g. 25s) to the `fetch()` call to OpenRouter in the build pipeline so a hang produces a `{"type":"error","message":"LLM timeout"}` event instead of silence.
4. If the Worker is hitting the CF wall-clock limit, verify the SSE keepalive ping is being sent — a stalled `fetch()` can eat the clock while the stream looks alive.

### P0 — Dashboard `/dashboard` returns 404 in production [REGRESSION — was GREEN last run]

`GET https://hswebappbuilder.space/dashboard` returns HTTP 404 with Next.js's built-in "This page could not be found" page. The dashboard route exists in code (`app/dashboard/page.tsx`), the frontend build compiles clean locally (0 TS errors, 6 routes), and the page is generated correctly in the local Next.js build. The 404 is served by the live Vercel deployment `dpl_BmNaxdmTmqKQmHeWVKcR6cqESRMp`.

**Root cause — Clerk middleware `auth.protect()` falling through to 404:**

`middleware.ts` calls `auth.protect()` on all non-public routes. For unauthenticated users this should redirect to `/sign-in`, but instead it renders a 404. This happens when `signInUrl` is not configured in the Clerk middleware and `NEXT_PUBLIC_CLERK_SIGN_IN_URL` is not set as a Vercel environment variable — Clerk cannot determine where to redirect, so the route appears as "not found."

The `DEV_BYPASS` path (`NEXT_PUBLIC_DEV_BYPASS_AUTH === '1'`) bypasses this — likely that env var was previously set in Vercel and has since been removed.

**Fix (two options — pick one):**

**Option A — Quick restore:** Add `NEXT_PUBLIC_CLERK_SIGN_IN_URL=/sign-in` to Vercel environment variables and redeploy. This gives Clerk the redirect target and restores the expected sign-in flow.

**Option B — Code fix (preferred):** Update `middleware.ts` to pass `signInUrl` explicitly:
```ts
const clerkAuthMiddleware = clerkMiddleware(async (auth, request) => {
  if (!isPublicRoute(request)) {
    await auth.protect({ redirectUrl: '/sign-in' })
  }
})
```
Then push + trigger Vercel redeploy.

**Check also:** Verify the Vercel dashboard that `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` and `CLERK_SECRET_KEY` are still set (the publishable key in the live HTML is `pk_live_Y2xlcmsuaHN3ZWJhcHBidWlsZGVyLnNwYWNlJA` — so the frontend key is intact; the middleware / server-side key may differ).
