### P0 — `/dashboard` returns 404 in production [NEW REGRESSION]

**Introduced by:** Vercel deployment on Mon 2026-06-08 09:41 UTC (confirmed via `last-modified` header).

**Root cause:** The June 8 Vercel deploy removed `NEXT_PUBLIC_DEV_BYPASS_AUTH=1` from the Vercel environment (correct direction for auth hardening), but did NOT add `NEXT_PUBLIC_CLERK_SIGN_IN_URL` to Vercel env vars. As a result, Clerk's `auth.protect()` in `middleware.ts` rewrites unauthenticated `/dashboard` requests to `/404` instead of `/sign-in`.

**Evidence:**
```
HTTP/2 404

x-clerk-auth-status: signed-out
x-matched-path: /404
last-modified: Mon, 08 Jun 2026 09:41:52 GMT
```

The route EXISTS in the codebase (`app/dashboard/page.tsx`) and builds clean locally (`npm run build` → 6 routes including `/dashboard` ✅). The problem is purely Vercel env config.

**Fix (Vercel dashboard — no code change needed):**
1. Go to Vercel → `hswebappbuilder.space` → Settings → Environment Variables.
2. Add `NEXT_PUBLIC_CLERK_SIGN_IN_URL` = `/sign-in` for all environments.
3. Trigger a Vercel redeploy (the route is already correct in the committed code).
4. Verify `curl -I https://hswebappbuilder.space/dashboard` returns `307 → /sign-in` (not 404).

> Note: If you need dashboard instantly without Vercel env access, add `NEXT_PUBLIC_DEV_BYPASS_AUTH=1` back to Vercel as a short-term bridge — but the correct long-term fix is the Clerk URL above.
