### P0 — `/dashboard` returns 404 in production (users locked out)

**Severity:** P0 — authenticated users cannot access the main app dashboard.

The route is code-correct (builds cleanly, TypeScript passes), but production returns 404 for unauthenticated requests to `/dashboard`. Root cause is almost certainly a Clerk auth/redirect misconfiguration in the production environment.

**Evidence:**
- `GET https://hswebappbuilder.space/dashboard` → HTTP 404 (probe confirmed)
- Local build: `app/dashboard/page.tsx` exists and generates `.next/server/app/dashboard.html` cleanly
- TypeScript: 0 errors; Next.js build: 6 routes including `/dashboard` — all pass

**Root cause:** `middleware.ts` calls `auth.protect()` on `/dashboard`. In production, if `NEXT_PUBLIC_CLERK_SIGN_IN_URL` is unset, empty, or set to a non-existent path, Clerk's redirect for unauthenticated users lands on a 404. This is an environment variable issue in the Vercel/hosting deployment, not a code bug.

**Fix:**
1. In Vercel project settings, verify these env vars are set for production:
   - `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` — valid live key (not test key)
   - `CLERK_SECRET_KEY` — valid live secret key
   - `NEXT_PUBLIC_CLERK_SIGN_IN_URL` = `/sign-in`
   - `NEXT_PUBLIC_CLERK_SIGN_UP_URL` = `/sign-up`
2. If keys are missing, copy from Clerk dashboard → publish a new Vercel deployment.
3. Also migrate `middleware.ts` → `proxy.ts` per Next.js 16 convention (see P3 below).
